# 模块 04 · 证据矩阵

> 第四步：每个判断需要什么证据先想清楚

## 铁律

```
NO EVIDENCE WITHOUT 12-FIELD SOURCING
```

任何不满足 12 字段（信源溯源 7 项 + 判断映射 5 项）的"证据"都是**未验证**，进"待补"清单，不进证据矩阵正文。**信源溯源 7 项含 v5 新增的"识别等级"字段**——本地资料若识别等级 ≠ "完整识别"，进"待复核"清单。

详见 `references/evidence_sourcing_protocol.md`（联网采集）与 `references/local_materials_protocol.md`（本地资料处理）。

违反铁律的证据将在模块 05 反方审稿被剔除，相关结论强制降档。

## 模块目的

把每个假设需要的证据系统整理成一张表。证据矩阵的核心价值在两栏并列的字段——**这条证据能支撑什么**、**这条证据不能支撑什么**。把"不能支撑什么"写清楚的这一半，往往比找到证据本身更难，也是专业研究机构专业性的体现。

## 典型触发场景

- 完成假设生成后进入第四步
- 用户已经有一堆资料，需要整理成结构化证据表
- 课题中期发现证据散乱，需要按假设重组
- 用户单模块调用："帮我做一份证据矩阵"或"帮我整理证据表"

## AI 能做什么

- 自动整理证据表的字段（假设 / 待验证事实 / 优先信源 / 证据等级 / 获取路径 / 能/不能支撑 / 缺口）
- 从原始材料里抽取关键事实点和数据
- 标注证据等级（一手公开数据 / 二手研报 / 行业访谈 / AI 二次概括）
- 提示证据缺口和需要补查的方向
- 对同一事实做多源交叉比对，标注口径差异

## AI 不替你做的

- **关键数据回原文** —— AI 二次概括的数据必须由研究员回原文核对
- **"不能支撑什么"的边界** —— 研究员对领域的理解决定证据边界写得对不对
- **证据等级的最终判断** —— 哪些信源在本课题语境下可信，研究员定

## 警示信号 · 你脑子里出现这些想法时停下

| 警示想法 | 反制 |
|---|---|
| "这个我记得是这样" | 必须联网搜索验证 |
| "大约 30%-40%" / "据估计" | 必须查精确数字 + 出处 |
| "业内普遍认为" | 必须给具体机构 + 报告 + 日期 |
| "之前查过类似的" | 必须重查（数字会变） |
| "这个搜索不出来，跳过吧" | 必须显式标"待补"，不能跳过 |
| "时间紧，先填着差不多的" | 填差不多的会传到模块 05 被剔除，更慢 |
| "用户的研究方向暗示这个数应该是 X" | AI 不应该顺着用户先验走，必须独立查 |

## 常见借口 · 反制表

| 借口 | 现实 |
|---|---|
| "差不多就行了" | 差不多 ≠ 证据 |
| "这是常识" | 常识需要出处证明 |
| "搜不到说明信息不存在" | 搜不到说明检索词不对，或数据不公开——不等于不存在 |
| "用现有知识能推出来" | 推出来 = 编造，禁止 |
| "12 字段太多，先填关键的" | 12 字段是唯一权威，缺一项即"未验证" |
| "这个案例是著名的，不用查" | "著名" ≠ 准确，案例细节经常被以讹传讹 |
| "时间紧，跳过联网搜索直接填" | 跳过的代价是模块 05 必改 + 结论强制降档，更慢 |

## 工作流

> **v6 主体声明自检**：执行本模块前，必须对照模块 01 的"研究主体声明"四字段（产品形态 / 主要受众 / 输出范围 / 输出禁区）做边界自检——任何动作若超出"输出范围"或落入"输出禁区"，立即停下并降档。


> **前置必读**：执行本模块前先读 `references/evidence_sourcing_protocol.md`——该协议是证据采集规则的唯一权威，本工作流引用它而不重复定义。

### 默认主路径：并行派多个事实核查员子代理（v7 升级 · v7.1 加硬配额）

**v7 核心改造**：把"1 个子代理处理所有假设"改为"**每个子判断派 1 个子代理并行处理**"。优势：

- **上下文聚焦**：每个子代理只看 1 个子判断 + 对应的正反假设，关键词矩阵设计更精准
- **多源交叉天然形成**：不同子代理走不同搜索路径，多源验证不需要强制规则，是结构副产品
- **失败隔离**：1 个子代理出问题不影响其他

**v7.1 核心改造（硬配额）**：每个子代理内部强制 6 项硬配额（**具体数值以 `subagents/fact-checker-prompt.md` "硬配额"节为唯一权威**——本文不复制数字，避免调参时改一处漏多处），未达配额不允许返回"信息不足"。这解决 v7 测试发现的"一两轮就放弃 + 信息少"问题。

**v7.2 补丁（本地资料豁免）**：内部数据 / 政策原文 / 客户内部材料主导的课题，本地资料若已"完整识别"且满足 Q4（≥ 5 条证据）+ Q5（≥ 2 条一手），联网部分 Q2/Q3/Q6 降为最低要求（Q2 ≥ 2 / Q3 ≥ 1 / Q6 ≥ 0）。子代理输出契约必须显式标"⚙️ 本地豁免模式"。

```
1. 主流程读取模块 02 子判断清单 + 模块 03 假设清单
2. 对每个核心子判断（默认全部 3-6 个），分别派 1 个事实核查员子代理
3. 每个子代理输入契约（详见 subagents/fact-checker-prompt.md）：
   - 研究主题
   - 模块 01 研究主体声明（4 字段）+ 边界声明
   - 本地资料登记（与该子判断相关的资料子集）
   - **专属于本子代理的 1 个子判断**：含正向假设 + 反向假设 + 验证事实清单 + 证伪条件
4. 多个子代理并行执行（Claude Code Agent 工具支持并发）
5. 每个子代理启动第一动作是"工具自检"，缺工具立即声明
6. 每个子代理按硬配额内部循环（最多 3 轮主题调整）执行检索（v7.1 新增）
7. 接收 N 个结构化输出（每个 工具状态 + 配额达成表 + 搜索关键词矩阵 + 12 字段表 + 待复核 + 待补 + 边界拒绝）
8. 主流程汇总 N 个输出：
   - 合并 12 字段证据矩阵（跨子判断的重复证据 → 保留多个版本标注来源子代理）
   - 合并待复核清单
   - 合并待补清单
   - 合并边界拒绝清单
   - 生成"工具状态汇总"——所有子代理的工具自检结果
   - **生成"配额达成汇总"**（v7.1 新增）——每个子代理的 6 项配额状态一目了然，⚠️/❌ 项研究员重点审视
9. 研究员审视并定稿——重点关注配额状态非 ✅ 的子判断，决定是否要求重派或接受配额部分达成
```

### 何时不用并行（兜底）

- **子判断数 ≤ 2** → 1 个子代理即可，并行收益不显著
- **研究员明示要求"简单跑一遍"** → 单子代理
- **Agent 工具受限 / 不可用** → 同会话执行（见下方兜底）
- **API 成本敏感场景** → 单子代理（v6 模式）

### 选择主路径 vs 兜底

| 选项 | 用什么 | 何时用 |
|---|---|---|
| **并行子代理**（v7 默认） | 每个子判断 1 个子代理 | 子判断 ≥ 3 + Agent 工具可用 |
| **单子代理**（v6 兼容） | 1 个子代理处理全部 | 子判断 ≤ 2 / 简单跑测 / 成本敏感 |
| **同会话兜底** | AI 主流程自己做证据采集 | 无 Agent 工具时 |

如选择并行或单子代理主路径 → 跳过下面"同会话执行流程"，直接进步骤 5（关键数据回原文核对）。

### 兜底 · 同会话执行流程（无 Agent 工具时）

如当前环境无 Agent 工具可用，或研究员明确要求同会话执行，按以下 7 步走：

**1. 拿假设清单做输入**

从模块 03 拿假设清单。如果是单模块调用，请用户给出假设或待验证事实清单。

**2. 展开证据需求**

对每个假设，AI 列出：

- 需要验证什么事实
- 优先去哪类信源找（招股书 / 监管披露 / 第三方平台数据 / 学术论文 / 行业访谈）
- 什么样的证据等级可以接受

**机构内部数据 vs 行业公开数据，前者优先级常常更高**——内部数据（本机构资源分布 / 已布局项目 / 历史经验 / 内部测算）直接决定"建议是否落到本机构能做的事"，而行业公开数据只能回答"行业大致是什么样"。研究员展开证据需求时，应主动识别哪些假设需要内部数据支撑，把内部数据缺口标为"关键缺口"——AI 给不出这类数据，但应主动提示研究员去取。

**3. 消化本地资料（v5 新增 · LOCAL FIRST-HAND BEFORE WEB SEARCH）**

调用模块 01 的"本地资料登记表"，按 `local_materials_protocol.md` 处理：

- 按形态调用对应工具（`anthropic-skills:pdf` / `:docx` / `:xlsx` / `:pptx` / Read）
- 提取关键内容并按 12 字段（含识别等级）填入证据矩阵
- 识别等级 ≠ "完整识别"的条目进"待复核"清单，不进证据矩阵正文
- 工具读取失败 → 识别等级"未识别"，提示研究员人工处理

**本地资料没消化完之前，不进入第 4 步联网采集**。

**4. 按协议采集证据（联网部分，补本地资料不足）**

完整执行 `evidence_sourcing_protocol.md`：

- **边界自检**（协议第二节）—— 对照模块 01 边界声明，落在排除范围的证据直接拒绝
- **联网采集**（协议第三节）—— 联网搜索 → 取原文 → 询问研究员，禁止跳步、禁止凭记忆
- **填 12 字段**（协议第四节）—— 信源溯源 7 项（含识别等级）+ 判断映射 5 项，缺一项视为"未验证"

工具不可用时按协议第三节(c)处理（显式标"未查证·待补"，禁止假装查过）。

**5. 关键数据回原文核对**

涉及关键判断的量化指标（毛利率、增速、规模、占比等）必须回原文核对。AI 二次概括的"大约在某区间"和原文披露的精确数据，可能差几个数量级，甚至方向相反。具体例子见 `references/cases/case_token_outbound.md` 第四步（76.1% → -0.4% 的经典对照）。

跨文档引用同一指标时做口径核对，信源标到章节 / 页码级别。

**6. 标"能/不能支撑"——证据范围判断**

> 注意：本步骤的"边界"指的是"证据本身的支撑范围"，**和步骤 4 的"边界自检"是不同概念**——后者指对照模块 01 的研究边界做剔除。

每条证据必须填"能支撑什么 / 不能支撑什么"两栏：

- 把假设里没覆盖的范围明确说出来（这条证据只覆盖云端，不覆盖私有部署）
- 把时间窗口的局限说出来（这条证据是 2024 数据，不能直接用于 2026 判断）
- 把样本局限说出来（这条证据来自一家头部厂商，不能推广到全行业）

写不出"不能支撑什么"，往往说明这条证据的支撑范围还没真想清楚。

**7. 汇总三类清单**

证据矩阵填完后，自动汇总三类清单（v5 调整）：

- **证据缺口清单** —— 哪些假设的证据还不充分，需要补查或下调结论强度
- **待复核清单** —— 识别等级 ≠ "完整识别"的本地资料条目 + 标 `[待边界确认]` 的条目 + 单一信源支撑关键判断的条目
- **边界拒绝清单** —— 边界自检拒绝的证据需求 + 工具未查到的事实

三类清单都是模块 05 反方审稿和模块 07 报告成稿的输入。

## 阶段产出格式

证据矩阵采用 **12 字段**格式（v5：新增"识别等级"列。详见 `evidence_sourcing_protocol.md` 第四节）。三个产出：

- **表 1：证据矩阵** —— 12 字段完整、识别等级"完整识别"的证据条目
- **表 2：待复核清单** —— 识别等级非完整 + 待边界确认 + 单一信源关键支撑等待研究员复核的条目
- **表 3：缺口清单** —— 边界拒绝 / 工具未查到 / 字段不全的条目

如需 Excel 形态，按上述 12 字段（信源溯源 7 项 + 判断映射 5 项）直接创建表头即可。

Markdown 形态（12 字段）：

```markdown
| 信源机构 | 发布日期 | URL | 访问日期 | 证据等级 | 识别等级 | 原文摘录 | 对应假设 | 待验证事实 | 能支撑什么 | 不能支撑什么 | 缺口 |
|---|---|---|---|---|---|---|---|---|---|---|---|
| ... | ... | ... | ... | ... | ... | ... | ... | ... | ... | ... | ... |
```

## 跨领域案例参考

不同研究类型的证据矩阵长得不一样——商业研究的证据多是市场数据和披露指标，政策研究的证据多是规则条款对照，技术研究的证据多是工程参数。读一个贴近你课题的案例：

- `references/cases/case_token_outbound.md` —— 商业研究证据矩阵（含 76.1% → -0.4% 这组"回原文"经典对照 · 当前仓库唯一完整案例）

政策 / 技术路线类证据矩阵案例暂未收录。核心纪律普适——任何形态都要"关键数据回原文，'能/不能支撑'两栏并列"。

无论哪种形态，核心纪律相同：**关键数据回原文，"能/不能支撑"两栏并列**。AI 给的中间结论会"差一点"，这一点点常常就是研究成不成立的分水岭。

## 与下游模块的衔接

证据矩阵完成后，进入**反方审稿**（模块 05）。审稿员主要做三件事：检查证据表和结论是否匹配；找证据不足、结论写满、口径不一致的地方；逐条检验"能/不能支撑"边界写得是否扎实。

证据矩阵填得越扎实，反方审稿能挑出的问题越少。


## 模块结束 · 硬停

<硬停>

**本模块到此结束。不要直接进入下一模块，即使研究员回复"继续""下一步""好"。**

### 输出顺序

1. 按"阶段产出格式"输出本模块的阶段产出
2. 列出"待研究员逐条确认的判断点"——直接来自本模块"AI 不替你做的"小节，把每条转成针对当前课题的具体问题
3. 停下，等待研究员逐条回应

### 反惯性规则

- 研究员的简短确认（"好""继续""OK""下一步""嗯""可以""go""next"）**不构成跳过本停止协议的授权**。遇到这类回应时，主动重述当前未确认的判断点，要求逐条回应
- 研究员若只回应部分判断点，对未覆盖项继续追问，不要默认其余项已通过
- 确认完成之前，不要预先起草、预览或暗示下一模块的内容

### 进入下一模块的条件

研究员对所有判断点逐条表态，或明确说出"全部认可，进入下一模块"——满足任一才能继续到 **模块 05 · 反方审稿**。

</硬停>
