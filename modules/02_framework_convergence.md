# 模块 02 · 框架收敛

> 第二步：从综述结构收敛到判断框架

## 工作纪律

任何框架若在领域常识层面站不住（产业链拆错、政策链拆错、技术链拆错），不能进入 模块 03 假设生成——必须回到本模块第 2-5 步重搭。本模块第 6 步"领域 30 秒回检"是这条纪律的执行点。

## 模块目的

把主问题拆成一组可验证的子判断，并把这些子判断压缩成一条研究员能用的判断框架。这一步决定了后面证据收集的方向——框架对了，证据就有归处；框架不对，证据再多也是散点。

AI 在这一步会很乐意给一个 8 章综述结构，看起来很全，综述感很强。本模块的核心动作是把这种综述结构收敛成判断框架。

判断框架不是只有一种形态。常见的有四种——**乘数公式、对比矩阵、决策树、因果链路**——每种形态适合不同类型的研究。详见 `references/judgment_framework_patterns.md`。在执行本模块前，先想清楚主问题适合哪种形态再去拆框架，能省下大量返工。

## 典型触发场景

- 完成问题定义后进入第二步
- 用户已经有主问题，但材料散，希望搭一个能用的判断骨架
- 用户已经在写报告，发现章节排布是综述式的，需要重新搭判断框架

## AI 能做什么

- 给多个框架备选（PEST / SWOT / 价值链 / 竞争结构 / 单位经济模型 / 决策树 / 矩阵 / 链路对比）
- 在每个维度下把可问的问题列全
- 提醒容易漏掉的变量（成本、竞争、合规、阶段判断、落地映射）
- 帮把零散的子判断按因果链或递进关系串起来
- 按主问题类型推荐合适的判断框架形态（公式 / 矩阵 / 决策树 / 链路）

## AI 不替你做的

- **框架能否回答主问题** —— 框架结构是否与主问题对得上
- **模块之间是否有重叠** —— 每个子判断是否相互独立
- **和委托方关切是否对得上** —— 框架的维度选择是否服务于决策

## 工作流

> **v6 主体声明自检**：执行本模块前，必须对照 模块 01 的"研究主体声明"四字段（产品形态 / 主要受众 / 输出范围 / 输出禁区）做边界自检——任何动作若超出"输出范围"或落入"输出禁区"，立即停下并降档。


**1. 拿主问题做输入**

如果走完整流程，从模块 01 拿到主问题。如果是单模块调用，请用户给出主问题——没有主问题就拒绝直接拆框架，往回走一步先做问题定义。

**2. 列候选框架（v6 强化 · 形态多样性自检）**

执行前先读 `references/consulting_frameworks_cheatsheet.md` —— 战略咨询常用框架按 4 类组织（产业链结构 / 演进阶段 / 市场竞争 / 切入决策），每个框架配适用场景 + 容易踩的坑 + 输出形态。

**v6 强制：不直接采用 模块 01 的预标形态**——重新审视主问题，独立选择形态。

**形态多样性自检（强制）**：

为当前主问题，**逐一**评估 4 种基础形态的适用性 + 优劣：

| 形态 | 适用？ | 为什么用 / 为什么不用 |
|---|---|---|
| **乘数公式** | ⬜ | "X 是否成立 / 可持续 / 真实"——多个必要条件并存时适用 |
| **对比矩阵** | ⬜ | "X 演变方向 / X 之间差异"——最不容易出错的形态 |
| **决策树** | ⬜ | "X 在我们条件下是否可行 / 选哪条路"——决策导向时适用 |
| **因果链路** | ⬜ | "X 对 Y 有什么影响 / Y 由什么导致"——影响传导分析时适用 |

**默认偏好简单形态**：对比矩阵 > 单一其他形态 > 复合形态。

> **复合形态不是"高级"——能用单一形态解决的问题，禁止用复合形态**。

按主问题类型从 cheatsheet 挑 1 个为主框架（必要时可叠加 1 层）。让研究员在候选里选 / 调整 / 推翻。

无现成框架贴合时，按主问题"要回答什么"自己搭一个原创框架——框架是为问题服务的。

### 警示信号 · 形态选择时停下

| 警示想法 | 反制 |
|---|---|
| "模块 01 预标了 X，那就用 X" | 模块 02 必须独立选择，不直接采用预标 |
| "这个问题既要判断必要性又要选路径，应该用复合形态" | 先看单一形态够不够——多数研究单一形态够了 |
| "决策树 + 乘数公式" 是首选组合 | 这是 AI 默认偏好，不是研究本身的需求 |
| "复合形态显得更专业" | 复合形态显得啰嗦，多增加证据采集负担 |

**3. 拆出子判断（一级）**

把选定框架的每个维度落成一个具体的、可以用证据回答的子判断。每个子判断必须满足：

- 是判断性问题（有立场、有边界、可被证据修正）
- 在现有信息条件下大致可证据化
- 对主问题有直接贡献

**4. 收敛主线（二级）**

子判断列完后，往下压一层——找出 1–3 条主线，把若干子判断按内在关系串起来。常见的主线类型：

- **因果链**（A → B → C）—— 适合"X 是否成立"的研究
- **演变线**（前 → 中 → 后）—— 适合政策 / 趋势研究
- **约束线**（X 限制 Y 限制 Z）—— 描述判断的反向边界
- **路径线**（节点 1 → 节点 2 → ...）—— 适合技术路线 / 决策研究

选 1–3 条最能讲清楚主问题的主线即可，不必凑数。商业可持续性研究里常见的搭配是"主因果链 + 约束线"两条；政策研究里常见的是单条"演变线"；技术研究里常见的是单条"路径线"。

**5. 压缩到紧凑形态（三级，可选但推荐）**

> 实操中，第 4 步主线和第 5 步紧凑形态经常**同步成型**——不必拘泥线性顺序。叠加框架的复杂研究里，主线和紧凑形态甚至是同一组动作的不同侧面。

把整个框架压成一个紧凑形态——研究员一眼能看穿全局判断的那种。形态选择参考 `references/judgment_framework_patterns.md`，按主问题类型选：

- **乘数公式** —— "X 是否成立 / 可持续 / 真实"（多个必要条件并存）
- **对比矩阵** —— "X 演变方向 / X 之间差异"
- **决策树** —— "X 在我们条件下是否可行 / 选哪条路"
- **因果链路** —— "X 对 Y 有什么影响 / Y 由什么导致"

不确定时先用对比矩阵——它是最不容易出错的形态，做到一半发现别的形态更合适，再迁移过去。

压不出紧凑形态时，至少把上一步的主线做扎实——主线本身就是研究员能用的最低标准的判断骨架。

**6. 领域专家 30 秒回检（关键纪律）**

框架起草完后、进入模块 03 之前，**强制进行一次领域回检**。AI 起草框架时凭印象会犯领域错误，最常见的是产业链拆解错位（典型案例：把"氢电耦合发电反送电网"当氢能产业主链，而真实主链是"绿电→制氢→化工转化→绿色制品"）。

回检至少问三件事：

- 产业链 / 政策链 / 技术链有没有拆错主线？
- 子判断有没有把"研究员感兴趣"误当成"委托方需要判断"？
- 每个子判断是否都有现实可获得的证据路径？

若回检发现框架与领域事实不匹配，先回到第 2-5 步重搭框架，**不要进入假设生成**。

## 阶段产出格式

按以下核心字段输出（直接在阶段产出文件里用 Markdown 写）：

```markdown
# [课题名] · 判断框架

## 主问题
[复用模块 01 的判断性主问题]

## 框架形态
- 选用形态：[乘数公式 / 对比矩阵 / 决策树 / 因果链路 / 混合]
- 选择理由：[为什么这个形态适合回答主问题]

## 一级子判断
1. [子判断 1]
2. [子判断 2]
3. [子判断 3]

## 二级主线
- 主因果链 / 演变线 / 约束线 / 路径线：[...]

## 紧凑表达
[公式、矩阵、决策树或链路图的文字版]

## 框架适配说明
- 与主问题的对应关系：[每个子判断对应主问题的哪一面]
- 与委托方关切的对应关系：[每个子判断对决策的价值]
- 已知遗漏的维度：[本框架明确不覆盖什么]
```

政策演变 / 技术路线类形态案例暂未收录，按 `judgment_framework_patterns.md` 的形态描述迁移即可。
